    内存数据库

    • 磁盘数据库
    • DBMS大部分的历史都是处理各种硬件限制
    • 最初数据库设计的时候，硬件是很不一样的
    • 单处理器(单核)，RAM严重受限，数据库必须存储在磁盘上，磁盘比现在慢很多
    • 但是现在的内存容量非常大，甚至数据都可以放到内存中
    • 通常结构化数据比较小，非结构化或者半结构化数据集比较大
    • 我们需要理解为什么传统面向磁盘的大缓存数据库无法获得最佳性能
    • 面向磁盘的数据库：数据库主要的存储位置是非易失性设备HDD或者SSD
    • 数据库通常被组织为固定大小的页面结构
    • 系统使用了缓存池来将磁盘中的数据缓存到内存中
    • 缓存中在磁盘和内存之间来回移动页面
    • 必须将元组记录的ID转换为内存中的位置
    • 工作线程必须将页面pin住，保证不会被换出到磁盘中
    • 并发控制：系统必须假设一个事务在读取不在内存中的数据时，可能在任何时候出现停顿
    • 日志和恢复：大部分数据库使用 STEAL + NO-Force缓存策略，修改必须被刷到WAL中
    • 日志包含了修改前后的记录
    • 有很多工作用来跟踪各种LSN
    • 各种机制的时间占比：BUFFER 34% LATCH 14% LOCK 16% LOG 12% BTREE 16% REAL WORK 7%

    • 论文：OLTP Through the Locking Glass, and What We Found There

    • 内存数据库

    • 假设主要存储的位置永久在内存中

    • 早期的想法在80年代提出，但是由于内存价格和容量，到现在变得易行
    • 第一个内存数据库在90年代发布，例如TimesTen，DataBlitz，Altibase
    • 数据组织
      • 不再按槽位式页面存储，但是仍然按页面组织
      • 直接通过内存指针访问 vs. 记录ID
      • 固定 vs. 可变数据池
      • 使用校验和从破坏的数据库中检测软件错误
      • 操作系统也是按页面组织内存
    • 索引：通常是数据库重启后进行重建，主要原因是避免日志开销
    • 查询处理
      • 顺序处理不再比随机处理快非常多
      • 传统的每次一条记录的迭代模型由于函数调用变得越来越慢
      • 这个问题在OLAP数据库中更严重
    • 日志和恢复
      • DBMS还需要WAL在非易失性存储上
      • 使用组提交来批量化刷新来摊销 fsync 开销
      • 由于没有脏页，所以不再需要各种LSN
    • 瓶颈
      • 如果IO不再是最慢的资源，大部分DBMS的架构由于其他瓶颈要做改变
      • 包括：Locking/Latching, Cache-Line Misses，指针追逐，谓词计算，网络
    • 并发控制
      • 隔离性
      • 对于内存数据库，事务锁的代价与访问内存相同
      • 新的瓶颈是事物访问相同数据导致的竞争
      • 数据库可以将锁信息与元组数据存放在一起：保证CPU缓存本地性
      • Mutex太慢了，需要使用CAS指令
      • 两节的锁（2PL）：死锁检测，死锁预防
      • 时间戳排序（T/O）：每次读写时检查冲突，将每次访问的元组拷贝保证重复读；
      • OCC：所有变更保存在私有空间中，在提交时检查冲突
    • 低冲突乐观协议，高冲突场景悲观协议
    • 各种并发协议的开销瓶颈
      • 锁抖动：2PL死锁检测，2PL WAIT_DIE
      • 时间戳分配: 所有时间戳协议 + WAIT_DIE
      • 内存分配：OCC + MVCC（不要使用malloc）
    • 思考：
      • 内存数据库设计与磁盘数据库有很大的不同
      • 世界终对内存数据存储和处理很满意
      • 相比SSD，DRAM的增长停滞不前
