    存储模型和数据布局

    数据组织

    我们可以认为内存数据库是一个很大的字节数组。

    • 数据库通过模式将字节转换为合适的类型。
    • 每个元组有一个头部包含了元数据。

    将元组存储为固定长度的数据可以很容易计算一个元组的起始位置点。

    数据表示

    • INTEGER/BIGINT/SMALLINT/TINYINT C/C++表示

    • FLOAT/REAL vs. NUMBERIC/DECIMAL IEEE-754标准/定点小数

    • TIME/DATE/TIMESTAMP 32/64比特从UNIX epoch开始的秒、毫秒、微秒

    • VARCHAR/VARBININARY/TEXT/CLOB

      • 如果类型大于等于64比特指向其它位置

      • 头部有长度和指向下一个位置的地址，接着是数据

    • 可变精度的数字类型，通常比任意精度的数字更快

    • 固定精度数字：NUMERIC/DECIMAL 需要额外的元数据保存

    • 可变长度字段，VARCHAR(128)

    • 字节对齐对性能有很大影响。（插入负载场景）

      1. 不对齐：0.523 MB/s
      2. 对齐加 padding：11.7 MB/s
      3. 对齐重新排序：814.8 MB/s

    存储模型

    行存： n元存储模型(NSM)

    1. DBMS 将一个元组的所有属性连续存放。
    2. 适合 OLTP 负载
    3. 使用每次一个元组的迭代模型
    4. 优势：更快的插入更新删除，对于需要整个元组的查询友好，可以使用面向索引的存储
    5. 劣势：对于仅扫描部分属性的查询不友好

    物理存储：堆表，索引表

    列存：分解存储模型(DSM)

    1. 所有元组的一个属性连续存放
    2. 适合 OLAP 负载
    3. 使用每次一个向量的迭代模型
    4. 优势：减少读取的大小，更好的压缩
    5. 劣势：对于点查，插入，插入，更新和删除都比较慢，原因是需要元组的分裂和拼接

    混合存储

    1. 当数据插入到数据库中时是热的(不久后被更新的概率大)
    2. 元组时间越长，更新越不频繁。
    3. 单个逻辑数据库实例，对热冷数据使用不同存储模型。
    4. 对于快速 OLTP 存储在NSM中。对于更有效的 OLAP 将数据迁移到DSM中。
    • 选择1：分离的执行引擎：镜像Oracle/IBM，Delta存储HANA
    • 选择2：单个灵活的架构：自适应存储

    系统目录

    1. 几乎所有数据库都是将数据库目录与通用数据以相同的方式存储
    2. DBMS需要感知事务，自动支持DDL和并发的事务
    3. 增列：NSM将元组拷贝到新区域，DSM增加新的列段
    4. 删列：NSM拷贝元组到新的区域，将列标记为废弃的，DSM删除列段即可
    5. 创建索引：扫描所整个表，填充索引，必须记录变更，扫描完成后锁表解决变更的记录
    6. 删除索引：逻辑删除索引，所有已经执行的事务还要能更新它
    7. 序列：与隔离保护的逻辑不同，回滚操作不会将序列回滚，所有插入操作时写写冲突

    思考

    我们已经放弃了混合存储模型

    1. 巨大的工程量
    2. Delta版本存储 + 列存基本是等价的

    系统目录非常难
