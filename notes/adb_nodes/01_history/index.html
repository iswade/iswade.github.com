    数据库历史

    • 正确、高性能代码；文档、测试；代码检视；大型代码库
    • 并发控制，索引，存储模型，压缩，并行JOIN，网络协议，日志与恢复，查询优化，执行，编译
    • 历史总是在不断地重复
    • 古老的数据库问题仍然跟今天相关
    • SQL vs. NoSQL
    • Relational vs. CODASYL
    • 关系模型总是会赢
    • 今天的数据库系统中很多思想并不是新的
    • 1960s
      • IDS：GE -> Honeywell，网络模型，一次一行记录
      • CODASYL：提出了程序访问数据库的标准，Charles Bachman，网络模型，一次一行记录
      • 网络模型：负载的查询，很容易损坏
      • IBM IMS: 阿波罗登月计划中用于跟踪购买清单的早期数据库系统，层次模型，程序员定义的物理存储格式，一次一行记录
      • 层次模型：重复数据，没有独立性
    • 1970s
      • Ted Codd: 观察到开发人员每次数据库模式、或者layout变更时都在重写IMS和Codasyl程序
      • 数据库抽象避免这样的维护：简单数据结构保存数据，通过高级语言访问数据，物理存储留给实现
      • 关系模型：A Ralational Model of Data for Large Shared Data Banks
      • System R; INGRES; Oracle
      • Jim Gray; Stonebraker; Ellison
    • 1980s
      • 关系模型赢了
      • DB2/Oracle/Informix/Sybase/Teradata/Ingres/InterBase
      • 对象模型：避免关系与对象的抗阻不匹配，将对象与数据库紧耦合，这些原始数据库基本都不存在了，但是许多技术以其他形式存在(JSON, XML)；查询复杂，没有标准API
    • 1990s
      • 无聊的日子；在数据库系统和应用负载方面没有什么进步
      • 微软从Sybase分叉后创建了SQL server
      • Postgres支持SQL语言
      • SQLite启动
    • 2000s
      • 互联网繁荣
      • 所有大的玩家都是重量级
      • 开源数据库丢失很多重要的特性
      • 很多公司定制自己的中间件来横向扩展单机数据库
      • 数据仓库
      • 分布式/无共享存储价格，关系型SQL，闭源
      • Netezza，Paraccel，monetdb，Greenplum，Datallegro, Vertica
      • NoSQL系统
      • 高可用和高扩展性
      • 无模式，无关系数据模型，文档，键值，无ACID，定制化的API，开源
      • HBASE，DynamoDB，MongoDB，NOSQL，CouchDB，Redis，Cassandra，RethinkDB，Couchbase，Neo4j，Riak
    • 2021s
      • NewSQL
      • 高性能OLTP负载，与NoSQL数据库系统一样，但是支持ACID
      • 关系型SQL，分布式，通常是闭源的
      • Dbshards, H-Store, Clustrix, CockroachDB, ScaleArc HANA，VoltDB，NuoDB，TiDB，Spanner，MemSQL，HyPer，YugaByte
      • HTAP
      • 与NewSQL系统一样执行快速OLTP，同时跟数据仓库系统一样执行OLAP查询
      • 分布式，无共享，关系型SQL，开源或者闭源
      • 云系统
      • 第一个DDaaS是已有DBMS的容器化
      • 有一些从头设计的新DBMS在云环境中运行
      • Snowflake，Redshift，Spanner，Aurora，Fauna，SlicingDice，DynamoDB，Microsoft
      • 共享磁盘引擎
      • 不再重新编写定制的存储管理器，直接利用共享存储
      • 可扩展的执行层，独立于存储
      • 支持日志结构化方法
      • Apache Drill, SpliceMachine, Snowflake, cloudra impala, Spark, HIVE, Presto
      • 这是大多数人在谈论数据湖时想到的
      • 图系统
      • 存储查询图数据，支持以图为中心的查询API
      • 最近的研究表明使用图为中心的执行引擎和存储管理，带来的收益还不确定
      • Neo4j, MemGraph, Dgraph, TigerGraph, JanusGraph, IndraDB，Giraph，graphbase.ai，TerminusDB
      • 时序数据库
      • 特定用于存储时序和事件数据的系统
      • 这些系统的设计有很强的假设：数据分布式和查询负载模式
      • M3，Timescale，influxdb，Vectoria Metrics，ClickHouse
      • 专用系统：嵌入式，多模，区块链、硬件加速
    • 思考：随着时间的推移，随着专用系统扩大其领域范围，DBMS类别的分界线将继续模糊。我相信关系模型和声明式查询语言促进了更好的数据工程
