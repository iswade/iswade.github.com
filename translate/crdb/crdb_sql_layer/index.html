    SQL 层

    SQL 层暴露了 SQL API 接口给开发，将 SQL 语句转换为 KV 操作。

    概述

    一旦部署了CockroachDB，开发人员只需要一个连接到集群的连接字符串和SQL语句即可开始工作。

    因为 CockroachDB 的节点的行为都是对称的，开发人员可以将请求发送给任何一个节点（这意味着 CockroachDB 可以与 LB 很好的配合工作）。不管哪个节点收到请求，都扮演了一个“网关节点”的角色， 因为有其它层处理请求。

    当开发者发送请求给集群时，以 SQL 的形式送达，但是数据最终从存储层被写入和读取都是 KV 对。为了处理这种场景，SQL 层将 SQL 语句转换为一个 KV 操作的计划，然后将其传递给事务层。

    组件

    关系结构

    开发人员以关系结构（即行和列）存储在CockroachDB中的数据。行和列的组合被组织到表中。表的集合被组织到数据库中。您的群集可以包含许多数据库。

    由于这种结构，CockroachDB提供了典型的关系特征，如约束（例如，外键）。这使应用程序开发人员可以相信数据库将确保应用程序数据的一致结构化;数据验证不需要单独构建到应用程序逻辑中。

    SQL API

    CockroachDB实现了ANSI SQL标准的很大一部分来显示其关系结构。您可以在此处查看 CockroachDB支持的所有SQL功能。

    重要的是，通过SQL API，我们还允许开发人员使用ACID语义事务，就像通过任何SQL数据库（BEGIN，END，ISOLATION LEVELS等）一样。

    PostgreSQL 协议

    SQL查询通过 PostgreSQL 协议到达您的集群。 这使得通过支持大多数与PostgreSQL兼容的驱动程序以及许多PostgreSQL ORM（例如GORM（Go）和Hibernate（Java）），可以简化将应用程序连接到集群。

    SQL 解析器，优化器，执行器

    在节点最终从客户端收到SQL请求后，CockroachDB会解析该语句，创建查询计划，然后执行该计划。

    解析

    接收的查询将根据我们的yacc文件（描述我们支持的语法）进行解析，并将每个查询的字符串版本转换为抽象语法树（AST）。

    逻辑计划

    AST随后分三个阶段转换为查询计划：

    1. AST被转换为高级逻辑查询计划。 在此转换期间，CockroachDB还执行语义分析，包括检查查询是否有效，解析名称，消除不需要的中间计算以及最终确定用于中间结果的数据类型。
    2. 简化逻辑计划，使用始终有效的转换进行优化。
    3. 使用搜索算法优化逻辑计划，该算法评估执行查询的许多可能方式并选择成本最低的执行计划。

    优化阶段的结果是优化的逻辑计划。这可以通过EXPLAIN观察到。

    物理计划

    物理规划阶段基于范围局部性信息决定哪些节点将参与查询的执行。 这是CockroachDB决定分发查询以执行接近数据存储位置的一些计算的地方。

    The result of physical planning is a physical plan and can be observed with EXPLAIN(DISTSQL).

    查询执行

    物理计划的组件被发送到一个或多个节点以供执行。在每个节点上，CockroachDB生成一个逻辑处理器来计算查询的一部分。节点内部或跨节点的逻辑处理器通过逻辑数据流相互通信。查询的组合结果将发送回收到查询的第一个节点，以便进一步发送到SQL客户端。

    每个处理器使用编码形式作为查询操纵的标量值。这是一种二进制形式，与SQL中使用的形式不同。因此，必须对SQL查询中列出的值进行编码，并且必须在将逻辑处理器发送回SQL客户端之前对逻辑处理器之间通信并从磁盘读取的数据进行解码。

    编码

    尽管SQL查询是用可解析的字符串编写的，但CockroachDB的较低层主要以字节为单位进行处理。 这意味着在SQL层，在查询执行中，CockroachDB必须将行数据从其SQL表示形式转换为字符串，并将从较低层返回的字节转换为可以传递回客户端的SQL数据。

    对于索引列而言，此字节编码保留与其表示的数据类型相同的排序顺序也很重要。 这是因为CockroachDB最终将数据存储在已排序的键值映射中; 以与其表示的数据相同的顺序存储字节使我们能够有效地扫描KV数据。

    但是，对于非索引列（例如，非PRIMARY KEY列），CockroachDB使用编码（称为“值编码”），它消耗较少的空间但不保留排序。

    您可以在编码技术说明中找到更详尽的详细信息。

    DistSQL

    由于CockroachDB是一个分布式数据库，我们为一些查询开发了一个分布式SQL（DistSQL）优化工具，可以大大加快涉及多个范围的查询。 虽然DistSQL的架构值得拥有自己的文档，但这个粗略的解释可以提供一些有关它如何工作的见解。

    在非分布式查询中，协调节点接收与其查询匹配的所有行，然后对整个数据集执行任何计算。

    但是，对于与DistSQL兼容的查询，每个节点对其包含的行进行计算，然后将结果（而不是整行）发送到协调节点。 然后，协调节点聚合来自每个节点的结果，最后向客户端返回单个响应。

    这大大减少了带到协调节点的数据量，并利用了久经考验的并行计算概念，最终减少了复杂查询完成所需的时间。 此外，它处理已经存储它的节点上的数据，这使得CockroachDB可以处理大于单个节点存储的行集。

    要以分布式方式运行SQL语句，我们引入了几个概念：

    • 逻辑计划：与上述AST / planNode树类似，它表示通过计算阶段的抽象（非分布式）数据流。
    • 物理计划：物理计划在概念上是逻辑计划节点到运行 CockroachDB 的物理机器的映射。 逻辑计划节点将根据群集拓扑进行复制和专用。 与上面的planNodes一样，物理计划的这些组件也在集群上进行调度和运行。

    您可以在 DistSQL RFC 中找到更多详细信息。
