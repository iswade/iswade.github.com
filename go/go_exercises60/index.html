    题19：关闭chan后读取

    func main() {
    var cc = make(chan int, 1)
    cc <- 5
    close(cc)
    for {
        select {
        case <-cc:
            fmt.Println("continue")
        case <-time.After(5 * time.Second):
            fmt.Println("timeout")
            return
        }
    }
}

    打印结果：

    死循环，不停地打印

    说明：

    对于通道c，内置函数 close© 记录该通道将不再发送任何值，如果继续发送则会产生panic。

    在调用close之后，并且在接收到任何先前发送的值之后，接收操作将返回通道类型的零值而不会阻塞。

    题28：map 并发安全

    下面关于Go语言map的说法，符合《华为Go语言安全编码规范》的是（C）

    1. A. 访问是并发安全的
    2. B. 并发写需要用锁，并发读不需要用锁
    3. C. 包内使用的map变量，如果只在init函数中修改，那么包内其他地方的读不需要加锁
    4. D. map并发访问导致的错误不影响运行，可以直接忽略
    func main() {
    var wg sync.WaitGroup
    m := make(map[int]string)
    wg.Add(3)
    l := &sync.Mutex{}
    go func() {
        l.Lock()      // 1
        m[1] = "one"
        l.Unlock()    // 1
        wg.Done()
    }()
    go func() {
        l.Lock()      // 2
        m[10] = "ten"
        l.Unlock()    // 2
        wg.Done()
    }()
    go func() {
        l.Lock()      // 3
        fmt.Println(m[10])
        l.Unlock()    // 3
        wg.Done()
    }()
    wg.Wait()
}
    1. 删除1，go run -race main.go，检测到数据竞争访问的错误
    2. 删除3，go run -race main.go，检测到数据竞争访问的错误
    3. 1，2，3全部保留，go run -race main.go，正确

    题30：接收器

    关于下面这段程序说法正确的是（C）

    type Transaction interface {
    SetTopTransType() bool
}

type scheduler struct {
    trans Transaction
}

func (s *scheduler) GetCurrentTransaction() Transaction {
    if s == nil {
        return nil
    }
    return s.trans
}
func (s *scheduler) GetCurrentTransactionType() bool {
    return s.GetCurrentTransaction().SetTopTransType()
}

func main() {
    s := scheduler{}
    s.GetCurrentTransaction()
    s.GetCurrentTransactionType()
}
    1. A. GetCurrentTransaction 中无需对s进行判空
    2. B. GetCurrentTransaction 有panic风险
    3. C. GetCurrentTransactionType 中有panic风险
    4. D. GetCurrentTransactionType 无panic风险

    题35：覆盖率

    关于Go语言测试覆盖率的描述，错误的是（B）

    1. A. 在go test中打开 -cover，-covermode, -coverpkg, -coverprofile中的任一选项，就可以打开测试覆盖率的功能
    2. B. go tool cover -func=coverage.data能输出函数中的每个语句的测试覆盖情况
    3. C. 覆盖率的统计模式有 set, count, atomic 三种
    4. D. 可以先在 go test 中增加 -coverprofile选项输出 coverage profile，再使用go tool cover工具具体分析每个函数的覆盖率

    covermode：

    • set: 每个语句是否执行
    • count: 每个语句执行了几次
    • atomic: 类似于 count, 但表示的是并行程序中的精确计数，性能差

    题38：import

    关于Go语言包的说法，错误的是（C）

    1. A. 包中引入 import 必须在程序的开头部分，package之后，方法和各类声明之前

    题60：编码规范

    对于下面的代码，符合华为《Go语言安全编码规范》的是（CE）

    func foo() {
    // 存放sessionID的map
    var sessionID = make(map[int]int)
    var mutex = &sync.Mutex{}
    var readOps uint64
    var writeOps uint64

    // 启动100个goroutine读取sessionID
    for r := 0; r < 100; r++ {
        go func() {
            total := 0
            for {
                key := rand.Intn(5)
                total += sessionID[key]
                atomic.AddUint64(&readOps, 1)
                time.Sleep(time.Millisecond)
            }
        }()
    }

    // 启动10个goroutine写入sessionID
    for w := 0; w < 10; w++ {
        go func() {
            key := rand.Intn(5)
            val := rand.Intn(100)
            mutex.Lock()
            sessionID[key] = val
            mutex.Unlock()
            atomic.AddUint64(&writeOps, 1)
            time.Sleep(time.Millisecond)
        }()
    }

    time.Sleep(time.Second)
}
    1. A. 变量readOps和writeOps没有初始化
    2. B. 读取和写入sessionID的goroutine个数必须相同
    3. C. total += sessionID[key]读取sessionID时，需要加锁确保并发安全
    4. D. atomic.AddUint64() 无法确保操作的原则性，需要加锁
    5. E. sessionID应该使用crypto/rand包生成安全随机数
